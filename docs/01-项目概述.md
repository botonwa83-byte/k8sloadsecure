# K8sGate - K8s 集群登录与审计系统

## 1. 项目背景

公司现有 K8s 集群（40 节点，3 主控），员工通过官方 Kubernetes Dashboard 6.1 进行日常操作。当前缺乏统一的登录认证、权限控制和操作审计能力。本项目旨在构建一个轻量级的登录与审计网关，在不改动现有 Dashboard 的前提下，实现用户认证、分级权限控制和操作日志记录。

## 2. 核心目标

- **统一登录认证**：账号密码登录，强制定期更换密码
- **分级权限控制**：Viewer（只读）、Developer（项目级读写）、Admin（管理员）
- **操作审计记录**：记录所有用户对集群的操作，支持按用户生成个人操作报告
- **零侵入**：不改动现有 K8s Dashboard，作为前置代理层运行

## 3. 现有环境

| 项目 | 详情 |
|------|------|
| K8s 版本 | v1.32.7 |
| 节点规模 | 40 个节点（3 主控 + 37 工作节点） |
| Ingress | Istio IngressGateway |
| 命名空间 | 约 60 个 |
| 现有 Dashboard | Kubernetes Dashboard 6.1（官方版） |
| 数据库 | MySQL 5.7 主从集群（已有） |
| 用户规模 | 约 500 人 |
| 操作系统 | Linux |

## 4. 技术选型

| 层 | 选型 | 理由 |
|----|------|------|
| 后端 | Go 1.22+ | 单二进制部署、client-go 原生支持 K8s、性能好、运维简单 |
| 前端 | Vue 3 + Element Plus | 轻量、组件丰富、Dashboard 式界面开发快 |
| 数据库 | MySQL 5.7 | 使用现有主从集群，不增加运维负担 |
| 部署方式 | K8s Deployment + Istio VirtualService | 适配现有 Istio 网关 |

## 5. 整体架构

```
用户浏览器
  |
  v
K8sGate（本系统）
  |-- 登录页 --> 认证 --> 签发 JWT
  |-- 管理后台（用户/项目/角色/审计报告）
  |-- 反向代理层 --+-- 拦截所有请求 --> 记审计日志
                   |
                   v
          K8s Dashboard 6.1（现有，不动）
                   |
                   v
            K8s API Server
```

### 5.1 请求流程

1. 用户访问 K8sGate 域名，未登录则跳转登录页
2. 输入账号密码，后端验证后签发 JWT Token（存储于 HttpOnly Cookie）
3. 用户进入管理界面，可选择进入 Dashboard 操作集群
4. 进入 Dashboard 时，后端根据用户角色选择对应权限的 ServiceAccount Token
5. 所有对 Dashboard/K8s API 的请求经过代理层，自动记录审计日志
6. 用户和管理员可在管理后台查看操作日志和个人报告
