## 1. 角色定义

### 1.1 系统角色（全局级别）

| 角色 | 说明 | 集群操作范围 | 管理后台权限 |
|------|------|-------------|-------------|
| **Admin** | 管理员 | 全部命名空间，全部操作 | 用户管理、项目管理、全部审计日志 |
| **GlobalViewer** | 全局只读用户 | 所有命名空间只读 | 查看所有审计日志和报告 |
| **Developer** | 项目开发者 | 对分配项目关联的命名空间读写 | 查看自己的操作日志和报告 |

### 1.2 角色权限对比

| 权限 | Admin | GlobalViewer | Developer |
|------|-------|--------------|-----------|
| 用户管理 | ✅ | ❌ | ❌ |
| 角色管理 | ✅ | ❌ | ❌ |
| 项目管理 | ✅ | ❌ | ❌ |
| 审批管理 | ✅ | ❌ | ❌ |
| 查看所有日志 | ✅ | ✅ | ❌ |
| 查看自己日志 | ✅ | ✅ | ✅ |
| K8s Dashboard | 完整 | 只读 | 受限 |

## 3. 权限控制实现

### 3.1 K8s RBAC 资源预创建

系统启动时，自动在 K8s 集群中创建以下 ClusterRole：

**只读 ClusterRole：**
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: k8sgate-viewer
rules:
  - apiGroups: ["", "apps", "batch", "networking.k8s.io"]
    resources: ["pods", "pods/log", "services", "deployments", "replicasets",
                "statefulsets", "daemonsets", "jobs", "cronjobs", "configmaps",
                "secrets", "ingresses", "endpoints", "events",
                "persistentvolumeclaims", "namespaces"]
    verbs: ["get", "list", "watch"]
```

**读写 ClusterRole：**
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: k8sgate-developer
rules:
  - apiGroups: ["", "apps", "batch", "networking.k8s.io"]
    resources: ["pods", "pods/log", "pods/exec", "services", "deployments",
                "replicasets", "statefulsets", "daemonsets", "jobs", "cronjobs",
                "configmaps", "secrets", "ingresses", "endpoints",
                "persistentvolumeclaims"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  - apiGroups: ["", "apps", "batch", "networking.k8s.io"]
    resources: ["namespaces", "events"]
    verbs: ["get", "list", "watch"]
```

**管理员：** 直接使用内置 `cluster-admin` ClusterRole。

### 3.2 动态权限绑定

用户登录后，后端根据角色动态处理：

**Admin 用户：**
- 使用预创建的 admin ServiceAccount（绑定 cluster-admin）
- 该 ServiceAccount 全局唯一，所有 Admin 共用

**Developer / GlobalViewer 用户：**
- 根据用户分配的项目，查出关联的命名空间列表
- 为每个命名空间创建 RoleBinding，将对应 ClusterRole 绑定到用户专属的 ServiceAccount
- ServiceAccount 按需创建，命名规则：`k8sgate-{user_id}`，创建在 `k8sgate-system` 命名空间

**示例：** 用户 zhangsan (id=1, role=developer) 分配了项目 A（命名空间 wfks, for-wfks）

```yaml
# 在 k8sgate-system 命名空间创建 ServiceAccount
apiVersion: v1
kind: ServiceAccount
metadata:
  name: k8sgate-1
  namespace: k8sgate-system

---
# 在 wfks 命名空间创建 RoleBinding
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: k8sgate-1
  namespace: wfks
subjects:
  - kind: ServiceAccount
    name: k8sgate-1
    namespace: k8sgate-system
roleRef:
  kind: ClusterRole
  name: k8sgate-developer
  apiGroup: rbac.authorization.k8s.io

---
# 在 for-wfks 命名空间创建 RoleBinding
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: k8sgate-1
  namespace: for-wfks
subjects:
  - kind: ServiceAccount
    name: k8sgate-1
    namespace: k8sgate-system
roleRef:
  kind: ClusterRole
  name: k8sgate-developer
  apiGroup: rbac.authorization.k8s.io
```

### 3.3 权限变更时的处理

当 Admin 修改用户角色或项目分配时：
1. 删除该用户旧的所有 RoleBinding
2. 根据新的角色和项目重新创建 RoleBinding
3. 使该用户的 JWT 失效（强制重新登录）

## 4. 代理层权限拦截

除了 K8s RBAC 本身的拦截外，代理层还做前置检查：

| 检查项 | 说明 |
|--------|------|
| JWT 有效性 | Token 是否过期、是否被吊销 |
| 密码过期检查 | 密码过期后只允许访问改密接口 |
| 命名空间白名单 | Developer/GlobalViewer 只能访问分配项目对应的命名空间，其他请求直接拒绝（不到 K8s API） |
| 操作类型检查 | GlobalViewer 的写操作在代理层直接拒绝 |

双重拦截（代理层 + K8s RBAC），确保即使代理层有 bug，K8s 自身 RBAC 仍然兜底。

## 6. 细粒度权限扩展（v2.0）

### 6.1 扩展角色体系

| 角色 | 描述 | 权限级别 |
|------|------|----------|
| super_admin | 超级管理员 | 全部权限 |
| admin | 管理员 | 管理权限 |
| security_admin | 安全管理员 | 审计、权限管理 |
| ops_admin | 运维管理员 | K8s资源管理 |
| global_viewer | 全局只读 | 查看所有资源 |
| project_admin | 项目管理员 | 项目内全部权限 |
| project_developer | 项目开发者 | 开发相关权限 |
| project_viewer | 项目查看者 | 只读权限 |
| project_operator | 项目运维 | 运维相关权限 |

### 6.2 角色继承关系

```
super_admin
```
