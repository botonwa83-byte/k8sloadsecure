# 审计与报告设计

## 1. 审计日志采集

### 1.1 采集方式

在 Go 后端的反向代理中间件中拦截所有请求，记录审计日志。

```
请求进入 --> JWT验证中间件 --> 审计日志中间件 --> 反向代理 --> K8s Dashboard
                                  |                              |
                                  |<---- 响应返回后补充状态码 ----|
                                  |
                                  v
                              写入 audit_logs 表
```

### 1.2 日志解析规则

从请求路径中解析资源信息，对应关系如下：

| 请求路径示例 | 解析结果 |
|-------------|---------|
| `/api/v1/namespaces/wfks/pods/web-abc` | namespace=wfks, resource_type=Pod, resource_name=web-abc |
| `/apis/apps/v1/namespaces/wfks/deployments/web` | namespace=wfks, resource_type=Deployment, resource_name=web |
| `/api/v1/nodes` | namespace=（空）, resource_type=Node |

### 1.3 操作摘要生成

对于写操作（POST/PUT/PATCH/DELETE），生成可读的操作摘要存入 detail 字段：

| 操作 | 摘要示例 |
|------|---------|
| 创建 Deployment | 创建 Deployment/web-server (副本数: 3) |
| 扩缩容 | 更新 Deployment/web-server 副本数: 3 -> 5 |
| 删除 Pod | 删除 Pod/web-server-abc123 |
| 更新 ConfigMap | 更新 ConfigMap/app-config |

### 1.4 异步写入

审计日志写入使用异步缓冲队列，避免影响请求响应速度：
- 内存中维护一个缓冲 channel（容量 1000）
- 后台 goroutine 批量写入 MySQL（每 100 条或每 2 秒写入一次）
- 系统关闭时 flush 剩余日志

## 2. 敏感操作识别

以下操作自动标记为敏感操作，在报告中高亮显示：

| 条件 | 说明 |
|------|------|
| action = DELETE | 所有删除操作 |
| resource_type in (Deployment, StatefulSet, Service, Ingress, PersistentVolumeClaim) 且 action != GET | 核心资源的写操作 |
| status_code = 403 | 被拒绝的操作（可能是越权尝试） |
| namespace in (kube-system, istio-system, monitoring) | 对系统级命名空间的写操作 |

## 3. 个人操作报告

### 3.1 报告内容

每个用户可查看自己的操作报告，包含以下模块：

**概览统计区：**
- 时间范围内总操作次数
- 按操作类型分布（GET/POST/PUT/PATCH/DELETE）
- 按结果分布（成功/失败/拒绝）
- 活跃天数
- 最常操作的命名空间 TOP 5

**操作明细表格：**
- 时间、操作类型、资源类型、资源名、命名空间、状态、操作摘要
- 支持按时间范围、操作类型、命名空间筛选
- 支持分页

**敏感操作列表：**
- 单独列出所有敏感操作
- 按时间倒序排列
- 标注敏感原因

### 3.2 报告查询 SQL 示例

**概览统计：**
```sql
-- 按操作类型统计
SELECT action, COUNT(*) AS cnt
FROM audit_logs
WHERE user_id = ? AND created_at BETWEEN ? AND ?
GROUP BY action;

-- 按结果统计（通过 status_code 判断）
SELECT
  SUM(CASE WHEN status_code BETWEEN 200 AND 299 THEN 1 ELSE 0 END) AS success,
  SUM(CASE WHEN status_code = 403 THEN 1 ELSE 0 END) AS denied,
  SUM(CASE WHEN status_code >= 400 AND status_code != 403 THEN 1 ELSE 0 END) AS failed
FROM audit_logs
WHERE user_id = ? AND created_at BETWEEN ? AND ?;

-- 活跃命名空间 TOP 5
SELECT namespace, COUNT(*) AS cnt
FROM audit_logs
WHERE user_id = ? AND created_at BETWEEN ? AND ? AND namespace != ''
GROUP BY namespace
ORDER BY cnt DESC
LIMIT 5;
```

### 3.3 导出 CSV

CSV 文件包含以下列：

```
时间,用户名,操作类型,资源类型,资源名,命名空间,状态码,客户端IP,操作摘要
2026-05-15 14:02:33,zhangsan,DELETE,Pod,web-abc,wfks,200,192.168.1.100,删除 Pod
```

## 4. 管理员全局报告

Admin 可查看全局统计，包含：

**用户活跃度排名：**
```sql
SELECT username, COUNT(*) AS cnt
FROM audit_logs
WHERE created_at BETWEEN ? AND ?
GROUP BY username
ORDER BY cnt DESC
LIMIT 20;
```

**命名空间操作热度：**
```sql
SELECT namespace, COUNT(*) AS cnt
FROM audit_logs
WHERE created_at BETWEEN ? AND ? AND namespace != ''
GROUP BY namespace
ORDER BY cnt DESC
LIMIT 20;
```

**异常操作汇总：**
- 被拒绝（403）操作次数和用户分布
- 非工作时间操作（可选，如 22:00 ~ 06:00 的操作）

## 5. 数据保留策略

| 数据 | 保留时长 | 处理方式 |
|------|---------|---------|
| audit_logs | 6 个月 | 超期数据定期归档到 audit_logs_archive 表，原表删除 |
| login_logs | 12 个月 | 超期数据直接删除 |

归档任务可通过 K8s CronJob 执行，每月 1 号凌晨运行：

```sql
-- 归档 6 个月前的审计日志
INSERT INTO audit_logs_archive SELECT * FROM audit_logs WHERE created_at < DATE_SUB(NOW(), INTERVAL 6 MONTH);
DELETE FROM audit_logs WHERE created_at < DATE_SUB(NOW(), INTERVAL 6 MONTH);
```
